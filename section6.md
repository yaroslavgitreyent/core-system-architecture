# Розділ 6. Архітектура системи та залежності між підсистемами

## 6.1. Загальний опис архітектури
Система "EcoSmart City" базується на сучасній трирівневій архітектурі (Three-Tier Architecture), що забезпечує гнучкість, масштабованість та легкість підтримки кожного окремого компонента.

### Рівні системи:
1. **Рівень представлення (Frontend):** - Веб-панель для адміністраторів та екологів (React/Vue.js).
   - Мобільний додаток для мешканців міста (Flutter/React Native).
   - Основне завдання: візуалізація даних та взаємодія з користувачем.

2. **Рівень логіки (Backend):** - Центральний API-сервер (Node.js або Python/FastAPI).
   - Сервіс авторизації (OAuth2/JWT).
   - Обробник потокових даних (Stream Processor) для прийому телеметрії від датчиків.

3. **Рівень даних (Database):** - PostgreSQL для зберігання структурованих даних (користувачі, пристрої).
   - InfluxDB або TimescaleDB для оптимізованого зберігання часових рядів (показники датчиків).

## 6.2. Взаємодія компонентів
Процес обміну даними відбувається за наступною логікою:
- **Датчики -> Сервер:** Дані передаються за протоколом MQTT або HTTP у форматі JSON.
- **Сервер -> База даних:** Після валідації дані записуються у відповідні таблиці.
- **Клієнт -> Сервер:** Користувач робить запит через REST API для отримання звіту або поточної карти.

## 6.3. Аналіз залежностей
- **Критична залежність:** Backend API не може функціонувати без бази даних.
- **Технічна залежність:** Точність аналітики екологів прямо залежить від стабільності роботи IoT-шлюзу, який агрегує дані з датчиків.
- **Користувацька залежність:** Мобільний додаток потребує постійного зв'язку з сервером для відображення push-сповіщень у реальному часі.
